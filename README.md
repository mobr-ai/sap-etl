# sap-etl

`sap-etl` is a C++ ETL pipeline that converts **native Jetstreamer local output** into RDF aligned with MOBR's Solana ontology and either:

- writes Turtle (`.ttl`) files, or
- sends batched SPARQL `INSERT DATA` updates to a QLever endpoint.

## Scope and reality constraints

This project is designed for **Jetstreamer’s outputs**.

Jetstreamer currently provides:

- historical replay by **slot range** or **single epoch**
- built-in ClickHouse-backed plugins including:
  - `program-tracking`
  - `instruction-tracking`
  - `pubkey-stats`
- a ClickHouse-backed server


Accordingly, `sap-etl` consumes **local ClickHouse tables created by Jetstreamer** over the ClickHouse HTTP interface. It does **not** parse internal ClickHouse data files directly, and it does **not** require any new Jetstreamer plugin.

## What `sap-etl` maps today

Because the built-in plugin outputs are aggregated tables, the RDF generated by `sap-etl` is limited to concepts that can be mapped from the available data. Currently the loader maps:

- `program_invocations`
- `slot_instructions`
- `jetstreamer_slot_status` when present

This means the current implementation emits RDF for:

- `so:Slot`
- `so:Block`
- `so:Program`
- selected telemetry-style ledger events grounded in native Jetstreamer aggregates

The codebase is structured so that future source adapters can add richer mappings when new native data sources become available.

## Features

- clean C++20 codebase with separation of concerns
- source / transform / load layering
- native ClickHouse-over-HTTP Jetstreamer source adapter
- TTL output with **automatic file rotation**
- QLever SPARQL update loading
- slot-range chunking
- checkpoint files for resume support
- parallel transformation with thread-local triple buffering
- tests with CTest
- no shell scripts required

## Repository layout

- `src/config` - CLI parsing and runtime configuration
- `src/source` - source adapters
- `src/clickhouse` - ClickHouse HTTP access
- `src/transform` - ontology mapping
- `src/load` - TTL and QLever loaders
- `src/checkpoint` - checkpoint persistence
- `src/model` - shared domain types and slot/epoch selection logic
- `tests` - unit tests

Install:

## Installing and running Jetstreamer

`sap-etl` expects **Jetstreamer-generated local data** to already exist.

### 1. Get Jetstreamer

Clone Jetstreamer from:

- `https://github.com/anza-xyz/jetstreamer`

### 2. Jetstreamer prerequisites

Jetstreamer documents the following setup characteristics that matter here:

- Rust toolchain / Cargo
- Clang **16** rather than 17 for RocksDB-related builds
- local ClickHouse helper support in the workspace

On Ubuntu / Debian, Jetstreamer documents installation using Clang 16 and supporting packages.

### 3. Build Jetstreamer

Inside the Jetstreamer repository:

```bash
cargo build --release
```

### 4. Run Jetstreamer for slots or epochs

Jetstreamer’s runner accepts either:

- a **single epoch**
- or a **slot range** written as `<start>:<end>`

Examples:

Epoch 900
```bash
cargo run --release -- 900
```

Slots from 388908000 to 388916000
```bash
cargo run --release -- 388908000:388916000
```

To enable the built-in native plugins that produce the local tables consumed by `sap-etl`, use:

```bash
cargo run --release -- 900 --with-plugin instruction-tracking --with-plugin program-tracking
```

Or for slot ranges:

```bash
cargo run --release -- 388908000:388916000 --with-plugin instruction-tracking --with-plugin program-tracking
```

### 5. Serve ClickHouse

```bash
cargo clickhouse-server
```

## How `sap-etl` reads Jetstreamer data

`sap-etl` connects to ClickHouse over HTTP, typically:

```text
http://127.0.0.1:8123
```

It reads native Jetstreamer tables and transforms the rows into RDF.

## Usage


### TTL output

TTL is the default output mode. To make it explicit:

```bash
./build/sap-etl \
  --load-type ttl \
  --scope slots \
  --slots 388908000-388916000
```


### Slot mode (default)

```bash
./build/sap-etl \
  --scope slots \
  --slots 388908000-388916000 \
  --clickhouse-url http://127.0.0.1:8123 \
  --clickhouse-database default \
  --output-dir ./out \
  --output-prefix sap-solana
```


### QLever output

```bash
./build/sap-etl \
  --load-type qlever \
  --scope slots \
  --slots 388908000-388916000 \
  --qlever-update-url http://127.0.0.1:7001
```


### Epoch mode

Slot mode is the default. Epoch mode is also supported, but because `sap-etl` is an external C++ process and not embedded in Solana runtime code, the epoch-to-slot conversion is driven by an explicit epoch schedule passed on the command line.

Example:

```bash
./build/sap-etl \
  --scope epochs \
  --epochs 800,801 \
  --slots-per-epoch 432000 \
  --first-normal-epoch 8 \
  --first-normal-slot 8160 \
  --warmup true \
  --clickhouse-url http://127.0.0.1:8123 \
  --output-dir ./out
```

Use the correct schedule for the ledger you replayed.

## Current options

- `--load-type ttl|qlever`
- `--scope slots|epochs`
- `--slots 1-10,20,30-40`
- `--epochs 800,801`
- `--clickhouse-url http://127.0.0.1:8123`
- `--clickhouse-database default`
- `--output-dir ./out`
- `--output-prefix sap-etl`
- `--checkpoint-dir ./checkpoints`
- `--ttl-max-size 50GiB`
- `--source-chunk-slots 10000`
- `--loader-batch-triples 20000`
- `--worker-threads 8`
- `--resume true`
- `--overwrite false`

## TTL rotation

Each TTL output file is capped by `--ttl-max-size`.

The default limit is **50 GiB**.

If the output grows past the limit, `sap-etl` automatically starts the next file:

- `sap-etl-0.ttl`
- `sap-etl-1.ttl`
- `sap-etl-2.ttl`

and so on.

## Checkpoint and resume

Checkpoints are written under `--checkpoint-dir`.

The current implementation stores the last fully completed slot chunk so interrupted runs can continue from the last durable boundary.

If you want a fresh run:

```bash
./build/sap-etl ... --resume false --overwrite true
```

## QLever notes

The QLever loader sends batches as SPARQL `INSERT DATA` updates over HTTP.

You are expected to point `--qlever-update-url` at a QLever endpoint that accepts SPARQL updates.

Example:

```bash
./build/sap-etl \
  --load-type qlever \
  --scope slots \
  --slots 388908000-388916000 \
  --qlever-update-url http://127.0.0.1:7001
```

## Extending the project

The design is ready for future native sources:

- add a new source adapter by implementing `JetstreamerSource`
- add richer ontology mapping in `OntologyMapper`
- keep the existing loaders unchanged

This preserves separation of concerns and avoids code duplication.
